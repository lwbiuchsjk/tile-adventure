### 术语约定（贯穿）

- **影响范围**：替代早期讨论中的"补给范围"。原因：游戏内既有"补给 = 移动力资源"的机制，沿用"补给"会混淆
- **补给**（旧义保留）：物品 / 移动力资源（对应代码中的 `ResourceType.SUPPLY`、`_supply`、`_camp_restore`），与"影响范围"无冲突
- **占据 / 驻扎**：占据 = 单位停留在 slot 格上；驻扎 = 占据的持续状态（回合末快照，离开即清零）

### 主项目 submodule 指针与本地 symlink 不一致

**现状**（2026-04-23 记录）：
- `tile-advanture-design` 在本地工作区是 symlink → `/mnt/d/坚果云/tile-advanture-design`（个人坚果云同步工作流）
- 主项目 git index 该 entry 仍是 gitlink（160000），指向 `6deb57d`
- `git status` 永久挂着 `typechange: tile-advanture-design`（未 stage）
- 远端 `origin/master` 的 submodule 指针仍是 `6deb57d`；submodule 仓库 `origin/main` 已到 `94db2f4`（含 M8 动态目标口径修正）
- 协作影响：其他人 clone 主项目 + init submodule 后，拿到的设计文档是 `6deb57d`，看不到 `94db2f4` 及之后的设计口径变更

**候选方案**（详见讨论 2026-04-23）：
- A 保持现状：本地 symlink 工作流不变，接受远端 submodule 指针落后；未来统一推一次指针
- B 临时恢复标准 submodule 形态 → 更新指针 → 换回 symlink（5 步手动操作，有副本与坚果云分叉风险）
- C 彻底放弃 submodule，stage typechange → tracked symlink；但破坏协作（协作者路径不存在）

**倾向**：B，但操作手动执行；需要时封装成 `tools/sync_design_submodule_pointer` 脚本。

**何时处理**：
- 近期：M9 或新设计文档有大改需同步给协作者时
- 或：有新协作者加入前必须处理
